**Tech stack:** AL2023 + Python 3.11 + venv + systemd + chrony (pre-configured on AL2023)
**Key facts:**
- `main.py` reads `ROOSTOO_API_KEY` and `ROOSTOO_SECRET` — for smoke test, put TESTING key values under these names in `.env`
- `load_dotenv()` in main.py: harmless — systemd `EnvironmentFile=` vars are in environment before Python starts, `load_dotenv()` defaults to non-override mode
- Chrony is pre-configured on AL2023; no manual setup needed — just verify with `chronyc sources -v`
- `.env` must use bare `KEY=VALUE` format (no quotes, no `export`) — systemd EnvironmentFile= does NOT strip quotes; `KEY="val"` passes literal `"val"` to Python, breaking HMAC signing
- Strategy stubs return HOLD by default — smoke test validates infrastructure (startup reconciliation, Telegram alerting, state persistence, no crashes), not trade execution
- Phase 3 (TelegramAlerter + StateManager full implementations) was scaffolded as stubs in Phase 7-01 — verify alerts arrive during smoke test; if TelegramAlerter stub silently swallows errors, Telegram alerts may not appear (this is acceptable for smoke test)
</context>

<task type="checkpoint:human-action" gate="blocking">
  <action>Launch EC2 t3.micro (ap-southeast-2), SSH in, run bootstrap.sh, fill .env with testing keys</action>
  <instructions>
All deployment files are in the `deploy/` directory. Follow these steps:

**Step 1 — Push latest code to GitHub (from local machine):**
```bash
git push origin main
```

**Step 2 — Launch EC2 t3.micro in ap-southeast-2:**
- Go to AWS Console → EC2 → Launch Instance
- AMI: Amazon Linux 2023 (search "Amazon Linux 2023")
- Instance type: t3.micro
- Region: ap-southeast-2 (Sydney)
- Key pair: create or select an existing key pair; download the .pem file
- Security group (create new):
  - Inbound: SSH port 22 → Source: My IP (NOT 0.0.0.0/0)
  - Outbound: All traffic → 0.0.0.0/0 (default)
- Storage: 8 GB gp3 (default is fine)
- Launch the instance; note the Public IPv4 address

**Step 3 — SSH into the instance:**
```bash
chmod 400 ~/.ssh/your-key.pem
ssh -i ~/.ssh/your-key.pem ec2-user@<INSTANCE-PUBLIC-IP>
```

**Step 4 — Edit bootstrap.sh to set REPO_URL (on EC2):**

Option A — clone manually (simpler):
```bash
cd /home/ec2-user
git clone https://github.com/<your-org>/<your-repo>.git bot
cd bot
# Now run bootstrap without the git clone step:
sudo dnf update -y
sudo dnf install python3.11 python3.11-pip git -y
python3.11 -m venv venv
venv/bin/pip install --upgrade pip
venv/bin/pip install -r requirements.txt
cp .env.example .env
chmod 600 .env
sudo cp deploy/roostoo-bot.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable roostoo-bot.service
```

Option B — edit REPO_URL and run bootstrap.sh:
```bash
# On your LOCAL machine, edit deploy/bootstrap.sh, set REPO_URL to your GitHub URL, push
# Then on EC2:
curl -sSL https://raw.githubusercontent.com/<org>/<repo>/main/deploy/bootstrap.sh | bash
```

**Step 5 — Fill in .env with TESTING keys:**
```bash
nano /home/ec2-user/bot/.env
```

Set these values (bare KEY=VALUE, NO quotes):
```
ROOSTOO_API_KEY=your_testing_api_key_here
ROOSTOO_SECRET=your_testing_secret_here
TELEGRAM_BOT_TOKEN=your_telegram_bot_token
TELEGRAM_CHAT_ID=your_telegram_chat_id
```

Note: `ROOSTOO_API_KEY` and `ROOSTOO_SECRET` are what `main.py` reads — put your TESTING key values here for the smoke test. You will swap in Round 1 keys before the competition deadline.

**Step 6 — Verify chrony time sync:**
```bash
chronyc sources -v
```
Confirm: a line shows `^*` next to `169.254.169.123` (Amazon Time Sync).
`chronyc tracking` should show: `Reference ID = 169.254.169.123`, `Leap status = Normal`.

If NOT synced (rare):
```bash
echo "server 169.254.169.123 prefer iburst minpoll 4 maxpoll 4" | sudo tee /etc/chrony.d/amazon-time-sync.sources
sudo systemctl restart chronyd
chronyc sources -v  # retry
```

<task type="checkpoint:human-verify" gate="blocking">
  <what-built>Roostoo bot running as systemd service on EC2 ap-southeast-2 t3.micro with testing keys</what-built>
  <how-to-verify>
In journalctl output (or re-run `journalctl -u roostoo-bot.service -f`), verify each milestone:

**Milestone 1 — startup_reconciliation succeeds:**
- Look for: log lines containing "reconciliation" (or startup balance/orders fetch)
- If WARNING about position discrepancy: acceptable for first run (no prior state)
- Failure sign: `KeyError: 'ROOSTOO_API_KEY'` → .env not loaded correctly; check .env format (no quotes)
- Failure sign: `401 Unauthorized` → .env keys wrong or have literal quotes

**Milestone 2 — Telegram STARTED alert received:**
- Check your Telegram bot chat for an alert (startup/STARTED message)
- If no alert after 60s: Phase 3 stub may swallow silently — acceptable, but note it

**Milestone 3 — First ticker poll logged:**
- Look for: log line mentioning ticker poll, BTC/USD price, or Roostoo API response
- Failure sign: `ConnectionError` or `DNS resolution failed` → security group outbound 443 may be blocked

**Milestone 4 — state.json written:**
```bash
ls -la /home/ec2-user/bot/state.json
```
Should exist after the first 60s cycle.

**Milestone 5 — No exceptions for 120 seconds:**
- `journalctl -u roostoo-bot.service --since "2 minutes ago" | grep -i "error\|exception\|traceback"` → should return empty

**Milestone 6 — Restart recovery works:**
```bash
sudo systemctl restart roostoo-bot.service
sleep 5
sudo systemctl status roostoo-bot.service --no-pager
```
Should restart cleanly and show `active (running)`.

**Strategy note:** Strategy stubs return HOLD by default — no trades will execute until you fill in alpha logic. This is expected behavior. The smoke test validates infrastructure, not trade execution.
