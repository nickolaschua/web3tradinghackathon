<task type="auto">
  <name>Task 2: Create scripts/download_data.py</name>
  <files>scripts/download_data.py</files>
  <action>
Create `scripts/download_data.py` — no `scripts/__init__.py` needed (not a
package, just a runnable script).

Script design:
- Use Binance public REST: `GET https://api.binance.com/api/v3/klines`
- No API key required (public endpoint, no auth headers needed)
- Pairs: BTCUSDT, ETHUSDT, SOLUSDT (all in 4h interval)
- Pagination: `limit=1000` per request (Binance max). Advance startTime
  to `last_row_open_time + 1ms` after each batch. Stop when batch returns
  < 1000 rows OR open_time >= end_ms.
- Sleep 0.1s between requests to stay well under Binance weight limits
  (weight budget is 1200/min; klines costs 2 weight per call)
- Default date range: `--start 2022-01-01` to today
- CLI args: `--start YYYY-MM-DD`, `--end YYYY-MM-DD`, `--output-dir data`

Response column mapping (Binance klines returns 12-element arrays):
  Index 0: open_time (ms)  → DatetimeIndex (UTC)
  Index 1: open            → float column "open"
  Index 2: high            → float column "high"
  Index 3: low             → float column "low"
  Index 4: close           → float column "close"
  Index 5: volume          → float column "volume"
  Indices 6-11: discard

Output:
- `output_dir.mkdir(parents=True, exist_ok=True)` before writing
- Save as `{output_dir}/{symbol}_4h.parquet` (e.g. `data/BTCUSDT_4h.parquet`)
- Final DataFrame: columns = [open, high, low, close, volume], dtype float64
- Index: DatetimeIndex named "timestamp", UTC timezone, from open_time col

Error handling:
- `resp.raise_for_status()` on each request
- Print progress: "Downloading BTCUSDT... 500 rows fetched..." per batch
- Print final: "BTCUSDT: 1523 rows saved to data/BTCUSDT_4h.parquet"
- If a pair fails, print error and continue to next pair (don't abort all)

DO NOT use `python-binance` or `binance-connector` — these add unnecessary
deps. Plain `requests` (already in requirements.txt) is sufficient.
DO NOT use `binance-historical-data` package — it downloads CSV from S3
and requires extra parsing; direct REST API is simpler and already understood.
